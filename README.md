# ⚡ NC Snapshot — Nine Chronicles Blockchain Snapshot Tool

Fast, production-grade snapshot tool for Nine Chronicles blockchain nodes, written in **Rust** with **C# bridge** integration for checkpoint validation and metadata generation.

> Replaces the [NineChronicles.Snapshot](https://github.com/planetarium/NineChronicles.Snapshot) C# tool with **~43% faster** performance.

---

## Features

- **🔥 Blazing Fast** — zstd multi-threaded compression (8 CPU cores) vs C# single-threaded zip
- **🔒 Integrity** — BLAKE3 checksums for every file, manifest embedded inside the archive (and saved externally)
- **📸 Live Snapshots** — Take consistent snapshots **without stopping the node** using validated RocksDB checkpoints
- **🧹 GC Pipeline (Prune)** — 4-phase garbage collection: **Rust export** → **Rust BFS** → **Rust prune** → **C# validation**
  - Reduces states/ by ~30% (66M kept vs 95M scanned)
  - Validated by Libplanet after pruning to guarantee consistency
- **📦 Partition Mode** — Creates **2 archives**: `partition/snapshot-{epoch}-{epoch}.tar.zst` (~300 MB) + `state/state_latest.tar.zst` (~58 GB)
- **🏷️ Auto-Naming** — Filenames automatically generated from bridge metadata (epoch-based)
- **📊 Progress** — Real-time progress bar with ETA and throughput
- **⚡ Atomic** — Writes to temp file, renames on success (no partial outputs)
- **📋 Metadata JSON** — Full metadata generated by the C# Bridge (same format as official tool)
- **🔍 Verification** — Verify archive integrity against embedded BLAKE3 manifest
- **🤖 JSON Output** — `--json` flag for scriptable automation
- **🛡️ Safe** — `9c-blockchain/` is **NEVER modified** — all checkpoints are created in a staging directory

## Quick Start

### State Snapshot (daily — offline)

```bash
./target/release/nc-snapshot create \
  --mode state \
  --apv "<APV>" \
  -s ~/9c-blockchain \
  --output-dir ~/snapshots/
# Creates: ~/snapshots/state/state_latest.tar.zst
```

### State Snapshot (live — node stays running)

```bash
./target/release/nc-snapshot create \
  --live \
  --mode state \
  --apv "<APV>" \
  -s ~/9c-blockchain \
  --output-dir ~/snapshots/
```

### Partition Snapshot (weekly — live + prune)

```bash
./target/release/nc-snapshot create \
  --live \
  --prune \
  --mode Partition \
  --block-before 0 \
  --apv "<APV>" \
  -s ~/9c-blockchain \
  --output-dir ~/snapshots/ \
  --level 1 \
  --epoch-validate 20638 \
  --epoch-limit 20640
# Creates: 
#   📦 partition/snapshot-20641-20641.tar.zst  (~192 MiB)
#   📦 state/state_latest.tar.zst              (~58 GiB)
#   📄 metadata/snapshot-20641-20641.json
```

### Verify Archive Integrity

```bash
./target/release/nc-snapshot verify ~/snapshots/state/state_latest.tar.zst
```

## Snapshot Modes

### `state` — State Snapshot (daily)
Contains indexes and state trie. Used to boot a new node quickly.
- `block/blockindex`, `tx/txindex` — block and transaction indexes
- `states` — full state Merkle trie (can be pruned with `--prune`)
- `chain`, `blockcommit`, `txexec`, `txbindex`

**Size:** ~68 GiB → ~58 GiB compressed (pruned states/: ~17-18.5 GiB)

### `partition` — Base/Partition Snapshot (weekly)
Creates **2 archives**:
1. **Partition archive** (`partition/snapshot-{epoch}-{epoch}.tar.zst`): block + tx epoch data (~300 MB)
2. **State archive** (`state/state_latest.tar.zst`): indexes + states + chain (~58 GiB)
3. **Metadata JSON** (`metadata/snapshot-{epoch}-{epoch}.json`): block headers

**Size:** ~394 MiB (partition) + ~68 GiB (state) compressed

### `full` — Full Snapshot
Everything in the store directory: all epochs, all indexes, all DBs. Single archive.

**Size:** ~400+ GiB

## Live Snapshots (`--live`)

The `--live` flag creates consistent snapshots **without stopping the Nine Chronicles node**.

### How It Works

1. **CheckpointBridge** (C#) creates validated checkpoints of each RocksDB:
   - Opens each DB in **secondary mode** (no LOCK conflict with running node)
   - Calls `rocksdb_checkpoint_create()` via native RocksDB C API
   - Validates the checkpoint by opening it with `Libplanet.RocksDBStore`
2. All epochs in `block/` and `tx/` are validated individually (batch mode)
3. The Rust tool archives **from the checkpoint** (not the live source)
4. Checkpoint is cleaned up after archiving

### Safety Guarantees

- **`9c-blockchain/` is NEVER modified** — all checkpoints are created outside it
- Validated by Libplanet — guarantees format_version compatibility
- `--block-before` (default=1) handles the tip offset for live snapshots
- If checkpoint creation fails, the tool exits with an error (never archives incomplete data)

### Epoch Validation (`--epoch-validate`)

For Partition/Full mode with thousands of epoch directories, use `--epoch-validate` to skip validating epochs below a certain number. This saves significant time:

```bash
--epoch-validate 20638  # Only validates epochs >= 20638 (saves ~10-15 min)
```

## GC Pipeline (Prune) — `--prune`

The GC Pipeline removes unreachable state trie nodes, reducing snapshot size by ~30%.

### 4-Phase Pipeline

```
Phase 1: Export (Rust 🦀)
  └─ Reads all KV pairs from states/ RocksDB → binary export file
  └─ Sequential scan, ~76 GB exported
  
Phase 2: BFS (Rust 🦀 — mmap-based)
  └─ Memory-maps the export file (zero-copy)
  └─ Level-by-level BFS traversal of Merkle trie
  └─ Finds all live (reachable) nodes
  └~ 33 levels, 66M live nodes found
  
Phase 3: Prune (Rust 🦀)
  └─ Copies only live keys to new states_gc/ DB
  └─ Format_version 5 (Libplanet compatible)
  └─ Write batches of 500K entries
  
Phase 4: Validate (C# — CheckpointBridge)
  └─ Opens pruned DB with Libplanet.RocksDBStore
  └─ Verifies integrity and compatibility
```

| Metric | Before | After |
|--------|--------|-------|
| Total nodes scanned | 94,991,843 | — |
| Live nodes kept | — | 66,285,258 |
| Garbage deleted | — | 28,706,565 (30.2%) |
| States/ directory | ~40-45 GiB | ~17-18.5 GiB |

```

├── state/
│   └── state_latest.tar.zst             ← indexes + states + chain
│   └── state_latest.tar.blake3          ← BLAKE3 manifest
```

## CLI Reference

```
nc-snapshot create [OPTIONS]

Options:
  -s, --source <PATH>           Blockchain directory [default: ~/9c-blockchain]
  -o, --output <PATH>           Output archive path — fixed name, overrides auto-naming
      --output-dir <PATH>       Output directory — file auto-named by epoch
  -m, --mode <MODE>             state | partition | full [default: state]
  -l, --level <1-19>            Zstd compression level [default: 1]
  -t, --threads <N>             Compression threads, 0=all CPUs [default: 0]
  -e, --exclude <DIR>           Directories to exclude (repeatable)
  -i, --include <DIR>           Override default dirs (repeatable)
      --epoch-limit <N>         Partition mode: archive epochs >= N
      --epoch-validate <N>      Checkpoint: only validate epochs >= N (faster)
      --apv <APV>               APV string for metadata generation
      --block-before <N>        Block before current tip [default: 1]
      --live                    Take snapshot without stopping node
      --prune                   Run GC Pipeline to remove dead state trie nodes
      --prune-depth <N>         State roots to preserve when pruning [default: 3]
      --force                   Proceed even if node is running
      --json                    JSON structured output
      --dry-run                 Scan only, don't create archive
      --incremental             Skip unchanged files (fingerprint-based)

nc-snapshot verify [OPTIONS] <ARCHIVE>
      --json                    JSON structured output
```

## Performance

### Latest Benchmark (2026-07-08 — Partition + Live + Prune)

| Metric | nc-snapshot (Rust 🦀) | NineChronicles.Snapshot (C# ☕) |
|--------|----------------------|--------------------------------|
| **Total time** | **1h 16m** | ~2h 14m |
| **Checkpoint** | ~20 min (validated) | ~15 min (validated) |
| **GC Pipeline** | ~10.7 min (Rust) | ~108.9 min (C#) |
| **Archive partition** | ~1.5s (192 MiB) | ~0.5 min |
| **Archive state** | ~482s (58 GiB) | ~14.5 min |
| **Compression** | zstd multi-threaded | zip single-threaded |
| **Integrity** | ✅ BLAKE3 checksums | ❌ None |
| **~43% faster** | 🚀 | |

### GC Pipeline: Rust vs C#

| Metric | Rust 🦀 | C# ☕ | Improvement |
|--------|---------|------|-------------|
| Prune total | ~10.7 min | ~14.7 min | **~37% faster** |
| Full Pipeline (Phases 1-4) | ~35 min* | ~109 min | **~3x faster** 🚀 |

\* *Phase 1 (Export) is I/O bound and similar between both implementations. Phase 2 (BFS) benefits from mmap-based zero-copy scanning in Rust. Exact per-phase benchmarks are pending.*

### Compression Levels

| Level | Speed | State Size (~68 GiB) | Time Est. |
|-------|-------|----------------------|-----------|
| 1 | ~145 MB/s | ~58 GiB | ~8 min |
| 3 | ~120 MB/s | ~55 GiB | ~10 min |
| 9 | ~40 MB/s | ~52 GiB | ~25 min |

Level 1 is recommended — RocksDB SST files are already internally compressed.

*Note: state archive size (~68 GiB uncompressed / ~58 GiB compressed) includes states/ + chain + blockindex + txindex + txbindex + blockcommit + txexec. The states/ directory alone is ~40-45 GiB before prune, ~17-18.5 GiB after.*

## Performance Evolution

A jornada de otimização do GC Pipeline, do C# ao Rust.

> **Setup:** Ryzen 2400G · 32 GB RAM · NVMe 1 TB  
> **Dataset:** states/ com 220 milhões de nós de uma blockchain que não rodava GC há 1 mês

### Fase 1: CopyStates C# — 24h 💀 (nunca terminou)

```
CopyStates(220M) → ⋯ → killed after 24 hours
```

A abordagem original do `NineChronicles.Snapshot` — copia **todos** os nós do state trie, sem poda. Com o node rodando por meses sem compactação, os 220M nós simplesmente não cabiam na janela de snapshot. O processo era **morto por OOM** após 24h.

| Nodes | Tempo | Memória | Resultado |
|-------|-------|---------|-----------|
| 220M → 220M | ∞ (killed) | 22-25 GB | ❌ Falhou |

---

### Fase 2: GC Pipeline C# (primeira limpeza) — 6h 🐌

```
Export(220M) → BFS(220M) → Write(66M)
```

**PRIMEIRA LIMPEZA DA HISTÓRIA!** O GC Pipeline em C# conseguiu o que o CopyStates não conseguia:
- **154 milhões de nós deletados** 🗑️ (70% de redução!)
- States/ foi de **~200+ GiB** (nunca limpo antes!) para **~20 GiB**
- Mas o BFS em C# carregava **tudo na RAM** — 22-25 GB de memória

| Nodes | Tempo | Memória | Resultado |
|-------|-------|---------|-----------|
| 220M → 66M | **~6h** | 22-25 GB | ✅ 154M deletados! |

---

### Fase 3: GC Pipeline C# (mantendo limpo) — 2h14m 🚗

```
Export(66M) → BFS(66M) → Write(66M)
```

Com o state trie já limpo (66M nós), o GC passou a rodar em **2h14m**. Apenas **28M novos nós de lixo** foram acumulados desde a última limpeza. Ainda consumindo 22-25 GB de RAM.

| Nodes | Tempo | Memória | Resultado |
|-------|-------|---------|-----------|
| 66M → 66M | **2h14m** | 22-25 GB | ✅ 28M lixo removido |

---

### Fase 4: GC Pipeline Rust (mantendo limpo) — 1h16m 🏎️

```
Export(Rust) → BFS-mmap(Rust) → Prune(Rust) → Validate(C#)
```

A virada! O Rust com **mmap-based BFS** (zero-copy) reduziu:
- **Tempo:** 2h14m → 1h16m (**~43% mais rápido**)
- **Memória:** 22-25 GB → 12-14 GB (**~45% menos RAM**)
- **Prune:** Em Rust puro 🦀, só a validação final em C#

| Nodes | Tempo | Memória | Resultado |
|-------|-------|---------|-----------|
| 66M → 66M | **1h16m** 🚀 | **12-14 GB** | ✅ 28M lixo removido |

---

### Fase 5: GC Pipeline Rust (primeira limpeza estimada) — ~3h 🚀

```
Export(220M) → BFS-mmap(220M) → Prune(66M) → Validate(C#)
```

**Se a primeira limpeza fosse em Rust** (estimado com base no throughput do mmap):

| Nodes | Tempo | Memória | Resultado |
|-------|-------|---------|-----------|
| 220M → 66M | **~3h** 🔥 | 12-14 GB | ✅ 154M deletados! |

Comparação direta com C#: **2x mais rápido e metade da RAM**.

---

### Timeline Resumida

```
Fase  | Implementação       | Nodes         | Tempo   | RAM       | Resultado
──────┼─────────────────────┼───────────────┼─────────┼───────────┼────────────
1️⃣    | CopyStates C# 💀    | 220M → 220M   | 24h     | 22-25 GB  | Killed
2️⃣    | GC Pipeline C# 🐌   | 220M → 66M    | ~6h     | 22-25 GB  | ✅ 154M deletados!
3️⃣    | GC Pipeline C# 🚗   | 66M → 66M     | 2h14m   | 22-25 GB  | ✅ 28M novo lixo
4️⃣    | GC Pipeline Rust 🏎️ | 66M → 66M     | 1h16m   | 12-14 GB  | ✅ 28M novo lixo
5️⃣*   | GC Pipeline Rust 🚀 | 220M → 66M    | ~3h*    | 12-14 GB  | ✅ 154M deletados!

*Fase 5 é estimada com base no throughput do mmap
```

### Por que Rust é mais rápido?

1. **mmap-based BFS** 🗺️ — Memory-mapped file = zero-copy scanning. O C# lê o arquivo exportado byte a byte na RAM. O Rust mapeia na memória virtual e deixa o kernel gerenciar.
2. **Menos RAM** 📉 — 12-14 GB vs 22-25 GB = quase metade.
3. **Prune nativo** 🦀 — Write batches otimizados, format_version 5, sem overhead de runtime .NET.
4. **Só validação em C#** — A fase mais crítica (validação Libplanet) ainda usa o C# testado e comprovado.

## Installation

### Prerequisites

- **Rust** 1.75+ (`curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh`)
- **.NET 6.0+ Runtime** (for the C# bridges)
- **Linux x86_64** (tested; arm64 may work)
- ~200 GB free disk (for checkpoint + archive creation)

### 1. Build the Rust Binary

```bash
git clone https://github.com/trincaxt/nc-snapshot
cd nc-snapshot
cargo build --release
# Binary: ./target/release/nc-snapshot
```

### 2. Publish the C# Bridges

```bash
# Metadata Bridge (from NineChronicles.Snapshot repo)
cd /path/to/NineChronicles.Snapshot
dotnet publish NineChronicles.Snapshot.Bridge \
  --configuration Release \
  --runtime linux-x64 \
  --self-contained false \
  -o /path/to/nc-snapshot/bridge-bin/

# Checkpoint Bridge (from checkpoint-bridge-for-rust/)
cd /path/to/nc-snapshot/checkpoint-bridge-for-rust
dotnet publish --configuration Release \
  --runtime linux-x64 \
  --self-contained false \
  -o ../bridge-bin/checkpoint/
```

```
│   ├── NineChronicles.Snapshot.Bridge    ← Metadata bridge (C#)
```

## Live Mode Decision Matrix

| Flags | Behavior |
|-------|----------|

| `--live` | Online snapshot with validated checkpoint |

| `--live --epoch-validate N` | Live snapshot, validates epochs >= N only |
